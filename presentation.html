1. Introduction to the Security Paradigm

HyperTrust abandons standard Role-Based Access Control (RBAC) in favor of Ciphertext-Policy Attribute-Based Encryption (CP-ABE). In a traditional web application, access is determined by a database lookup (e.g., if user.role == "admin": grant_access()). In HyperTrust, access configurations are built deterministically into the cryptography itself.

If a user does not mathematically satisfy the cryptographic policy string (e.g. (dept:cse and paid:true) or role:networkadmin), their private key will physically fail to decrypt the sensitive payloads (like the Wi-Fi Token) or deny them access to secure portals.

2. Deep Dive: `abe_engine.py` (The Cryptographic Simulation)

Real-world CP-ABE uses elliptic curve bilinear pairings resulting in massive computational overhead. For HyperTrust's web scale, we developed a pure-Python structural simulation that exactly replicates the logical properties of ABE using symmetric cryptography (HKDF & AES logic).

Setup() & KeyGen()

- cpabe_setup() initializes the system mathematically. It generates a public key parameter PK and a Master Secret Key MSK. The MSK is never exposed.
- cpabe_keygen() is called when an Admin creates a User. It bounds a list of attributes to a specific ID. Crucially, we hash a unique user_salt per user. 

# The Collusion Barrier inside abe_engine.py:
user_salt = hashlib.sha256(f"user-salt-{user_id}-{pk['pk_seed']}".encode()).digest()

The user_salt ensures Collusion Resistance. If Alice has dept:cse and Bob has paid:true, they mathematically cannot merge their keys to bypass an AND requirement because the underlying derived hashes will reject the mismatched salts.

Encrypt() & Decrypt()

- cpabe_encrypt() generates an AES policy_key. It recursively scans the boolean policy string and creates a secure XOR-padded cipher share for each leaf attribute utilizing the PK.
- cpabe_decrypt() traverses the tree computationally. If the user's specific attribute subsets mathematically satisfy the tree logic, the original policy_key falls out, unlocking the cipher.

3. Deep Dive: Hybrid Cryptography (`crypto_utils.py`)

ABE blocks contain size constraints. To encrypt a Wi-Fi string or arbitrary HTML tokens, HyperTrust utilizes Hybrid Cryptography combining CP-ABE with AES-GCM 256.

The Encrypt Flow:

  1. An AES-256 Symmetric Key is randomly generated bytes.
  2. The plaintext (e.g., the "WIFI_XYZ" string or dummy "PORTAL_ACCESS" verification string) is encrypted using AES-GCM (which provides both secrecy and authenticity).
  3. The actual AES-256 Symmetric Key is then encrypted using cpabe_encrypt() under the specified policy matrix.
  4. The system stores the bundle: encrypted_token, nonce, tag, encrypted_aes_key

4. Web Integration (`routes/user.py` & Portals)

The HyperTrust architecture proves its cryptographic guarantees by actively gating the application routing.

When a user clicks "Request WiFi Access", the back-end dynamically generates a temporary WiFi string and wraps it in a hybrid ABE cipher utilizing the global WIFI_POLICY. The server attempts to use the requesting user's JSON private key to decode it. Only a mathematical success grants the Wi-Fi.
